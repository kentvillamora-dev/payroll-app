# Ourus — Identity & Access Schema
**Document 1 of 2 | Layer: Who users are and what they're allowed to do**

## What This Document Covers

This document covers the first layer of the Ourus database: identity and access control. Think of this layer as the **security gate** of the system — it answers two questions before anything else happens:

1. **Who are you?** (authentication — handled by Supabase)
2. **What are you allowed to do?** (authorization — handled by these tables)

There are **5 tables** in this layer. Run them in the order they appear below, because some tables reference others (just like how you can't write a VLOOKUP formula before the source sheet exists).

---

## Tables Overview

| # | Table | Plain-English Purpose |
|---|---|---|
| 1 | `platform_users` | You and your dev partners — platform-wide super admins |
| 2 | `companies` | Your client businesses |
| 3 | `users` | Employees of those client businesses |
| 4 | `roles` | Permission sets defined per company during onboarding |
| 5 | `user_roles` | Which employee has which role (the junction/bridge table) |

## How to Read the SQL Below

- **`--`** means a comment. SQL ignores everything after `--` on that line. These are notes for you.
- **`UUID`** is a randomly generated unique ID (looks like `a3f2c1d0-...`). Supabase generates these automatically.
- **`REFERENCES`** is the SQL word for "this column must match a value that already exists in that other table" — exactly like data validation in Excel, but enforced by the database itself.
- **`NOT NULL`** means the column cannot be left blank — like a required field in a form.
- **`DEFAULT`** means "use this value automatically if nothing is provided."
- **`UNIQUE`** means no two rows can have the same value in this column.

## Table 1 — `platform_users`

**What it is:** The roster of you and your development partners. This table is completely isolated from all client data. No client company, no employee role — just platform-level access. This is the Zero Trust principle in action: platform access and tenant access are structurally different things living in different tables.

```sql
-- Create the platform_users table.
-- This stores YOU and your dev partners only.
-- It is completely separate from client employee data.
CREATE TABLE platform_users (

  -- The unique ID for each platform user.
  -- "uuid" means it's a long random string like "a1b2c3d4-..."
  -- "PRIMARY KEY" means no two rows can ever have the same id.
  -- "DEFAULT gen_random_uuid()" means Supabase generates this automatically — you don't type it yourself.
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- This links to Supabase's built-in login system.
  -- When someone signs in via Supabase Auth, they get an auth ID.
  -- We store that auth ID here so we know which Supabase login belongs to which platform user.
  -- "REFERENCES auth.users(id)" means: this value MUST exist in Supabase's auth table first.
  -- "ON DELETE CASCADE" means: if the Supabase auth account is deleted, this row is also deleted automatically.
  auth_user_id UUID NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,

  -- The person's full name. Cannot be left blank.
  full_name TEXT NOT NULL,

  -- Their email address. Must be unique — no two platform users can share an email.
  email TEXT NOT NULL UNIQUE,

  -- Whether this account is currently active.
  -- FALSE means disabled (e.g., a former dev partner). The row stays for audit history.
  -- DEFAULT TRUE means new accounts start as active unless you say otherwise.
  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  -- The timestamp when this row was created.
  -- "TIMESTAMPTZ" means it stores the date, time, AND timezone.
  -- "NOW()" means "use the current date and time automatically."
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- The timestamp of their most recent login. Can be NULL if they haven't logged in yet.
  last_login_at TIMESTAMPTZ
);

-- Add a description to the table so future developers understand its purpose.
COMMENT ON TABLE platform_users IS
  'Ourus platform staff only (owners and dev partners). Completely isolated from client/tenant data.';
```

## Table 2 — `companies`

**What it is:** Your client masterlist. Every business that subscribes to Ourus gets one row here. All employee data eventually traces back to this table — it's the anchor of the entire multi-tenant design.

```sql
-- Create the companies table.
-- One row = one client business that uses Ourus.
CREATE TABLE companies (

  -- Unique ID for each company. Auto-generated by Supabase.
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- The official business name. Cannot be blank.
  name TEXT NOT NULL,

  -- The company's email domain (e.g., "sunrisebpo.com.ph").
  -- Used to validate that invited employees are using the correct corporate email.
  -- UNIQUE means no two companies can share the same domain.
  email_domain TEXT NOT NULL UNIQUE,

  -- Physical address. Optional — hence no NOT NULL.
  address TEXT,

  -- The subscription tier this company is on.
  -- CHECK constraint works like Excel data validation — only these exact values are allowed.
  subscription_plan TEXT NOT NULL DEFAULT 'STARTER'
    CHECK (subscription_plan IN ('STARTER', 'PROFESSIONAL', 'ENTERPRISE')),

  -- Whether this company's subscription is currently active.
  -- FALSE means their employees cannot log in.
  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  -- The country code. Defaults to PH since your primary market is the Philippines.
  country_code CHAR(2) NOT NULL DEFAULT 'PH',

  -- Timestamp of when this company was onboarded onto Ourus.
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

COMMENT ON TABLE companies IS
  'One row per client business subscribed to Ourus. All tenant data traces back to this table.';
```

## Table 3 — `users`

**What it is:** The identity record for every employee across all client companies. This is the most connected table in the identity layer — it links to Supabase Auth (for login), to `companies` (for tenancy), and will later be referenced by timekeeping and payroll tables.

```sql
-- Create the users table.
-- One row = one employee at one client company.
CREATE TABLE users (

  -- Unique ID for each employee record. Auto-generated.
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- Links to Supabase's login system — same pattern as platform_users.
  -- Every employee has a Supabase login; this column connects it to their profile here.
  auth_user_id UUID NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,

  -- Links this employee to their company.
  -- "REFERENCES companies(id)" means: the company must already exist in the companies table.
  -- This is the column that enforces "one employee = one company" — Philippine labor law by design.
  company_id UUID NOT NULL REFERENCES companies(id) ON DELETE RESTRICT,

  -- The 6-digit employee ID your document specifies.
  -- Stored as TEXT because leading zeros matter (e.g., "000123") and we won't do math on it.
  employee_id VARCHAR(6) NOT NULL,

  -- Full name of the employee.
  full_name TEXT NOT NULL,

  -- Work email address.
  email TEXT NOT NULL UNIQUE,

  -- Whether this account is active. FALSE = offboarded employee. Row stays for payroll history.
  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  -- Timestamp of when this employee was onboarded.
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- Who invited this employee. Links back to this same table (a supervisor or HR admin).
  -- NULL is allowed because the very first admin has no "inviter."
  invited_by UUID REFERENCES users(id) ON DELETE SET NULL,

  -- UNIQUE constraint: employee_id must be unique WITHIN a company, but two different
  -- companies can both have an employee "100001". This is like a composite key in Excel.
  CONSTRAINT unique_employee_id_per_company UNIQUE (company_id, employee_id)
);

COMMENT ON TABLE users IS
  'One row per client-side employee. Scoped to one company. Referenced by timekeeping and payroll layers.';
```

## Table 4 — `roles`

**What it is:** The permission sets you define for each company during the onboarding consultation. Roles are company-scoped — "HR Manager" at Sunrise BPO is a completely separate role from any similar title at Davao Logistics. Only you and your dev partners create rows here. Company admins only assign them.

```sql
-- Create the roles table.
-- One row = one permission set for one specific company.
CREATE TABLE roles (

  -- Unique ID for each role. Auto-generated.
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- Which company this role belongs to.
  -- A role cannot exist without a company — NOT NULL enforces this.
  company_id UUID NOT NULL REFERENCES companies(id) ON DELETE CASCADE,

  -- Human-readable name shown in the UI (e.g., "HR Manager", "Team Leader").
  role_name TEXT NOT NULL,

  -- Short code used in application logic (e.g., "HR_MGR", "TL", "AGENT").
  -- Think of this as the lookup key — your code checks role_code, not role_name,
  -- because role_name might change but role_code stays stable.
  role_code TEXT NOT NULL,

  -- Plain-English description of what this role can do.
  description TEXT,

  -- Who created this role — links to platform_users (only devs create roles).
  created_by UUID REFERENCES platform_users(id) ON DELETE SET NULL,

  -- Timestamp of creation.
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- Ensures no duplicate role codes within the same company.
  -- Two companies CAN have the same role_code — that's intentional.
  CONSTRAINT unique_role_code_per_company UNIQUE (company_id, role_code)
);

COMMENT ON TABLE roles IS
  'Company-scoped permission sets. Defined by Ourus devs during consultation. Assigned by company admins.';
```

## Table 5 — `user_roles`

**What it is:** The junction (bridge) table that connects employees to their roles. In spreadsheet terms, this is the many-to-many bridge — like a SUMIFS table where you look up by both `user_id` and `company_id` to get the full picture. One employee can have multiple roles simultaneously.

```sql
-- Create the user_roles table.
-- One row = one role assignment for one employee.
-- An employee with two roles will have two rows here.
CREATE TABLE user_roles (

  -- Unique ID for each assignment record.
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  -- Which employee is being assigned a role.
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

  -- Which role is being assigned.
  role_id UUID NOT NULL REFERENCES roles(id) ON DELETE CASCADE,

  -- Safety check: the role being assigned must belong to the same company as the user.
  -- This prevents a bug where a user could accidentally be assigned a role from a different company.
  company_id UUID NOT NULL REFERENCES companies(id) ON DELETE CASCADE,

  -- Who made this assignment — the company admin who onboarded the employee.
  assigned_by UUID REFERENCES users(id) ON DELETE SET NULL,

  -- Timestamp of when this role was assigned.
  assigned_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- Whether this role assignment is currently active.
  -- FALSE = role was revoked but kept for audit history.
  is_active BOOLEAN NOT NULL DEFAULT TRUE,

  -- Prevents the same role from being assigned to the same user twice.
  CONSTRAINT unique_user_role UNIQUE (user_id, role_id)
);

-- This is a database-level safety rule (called a "policy" or "constraint").
-- It ensures the role being assigned actually belongs to the same company as the user.
-- Think of it as a cross-sheet validation in Excel — both lookups must agree.
ALTER TABLE user_roles
  ADD CONSTRAINT role_belongs_to_company
  FOREIGN KEY (role_id, company_id)
  REFERENCES roles(id, company_id)
  -- Note: for this constraint to work, roles needs a unique index on (id, company_id).
  -- Run this line too:
  -- CREATE UNIQUE INDEX roles_id_company_id ON roles(id, company_id);
  DEFERRABLE INITIALLY DEFERRED;

COMMENT ON TABLE user_roles IS
  'Junction table: connects employees to their roles. Includes company_id as a structural safety check.';
```

> **Note for Supabase:** After running the `user_roles` table SQL, also run this line separately in the Supabase SQL editor:
> ```sql
> CREATE UNIQUE INDEX roles_id_company_id ON roles(id, company_id);
> ```
> This enables the cross-company safety check on the last constraint above.

## Recommended Run Order in Supabase

Go to your Supabase project → **SQL Editor** → run each block in this order:

1. `companies` — no dependencies
2. `platform_users` — depends on Supabase Auth only
3. `users` — depends on `companies` and Supabase Auth
4. `roles` — depends on `companies` and `platform_users`
5. `user_roles` — depends on `users`, `roles`, and `companies`
